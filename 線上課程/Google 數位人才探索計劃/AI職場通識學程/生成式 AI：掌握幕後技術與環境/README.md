# 生成式 AI：掌握幕後技術與環境

> 生成式 AI 領袖學習路徑 · 第三堂課學習筆記

## Insight

生成式 AI 不是只有聊天機器人——**應用程式只是冰山一角**。領導者必須理解五大層面（基礎架構 → 模型 → 平台 → 代理 → 應用程式）如何串連，才能在「算力／模型／工具／人」之間做出正確取捨。**代理 ≠ 模型**：代理 = 模型 + 推論迴圈 + 工具；**MLOps ≠ AutoML ≠ Model Garden**，三者分別管生命週期、低門檻建模、現成模型探索。專案規劃時，**自訂程度越高，人力、成本、時間越高**；緊急任務優先選現成應用程式，而非微調模型。

## Why

| 以前覺得 | 現在覺得 |
|----------|----------|
| 導入 AI = 選一個聊天機器人或 API。 | 要先問：算力夠嗎？用現成模型還是自訓？要代理還是單次生成？部署在雲端還是邊緣？ |
| Agent 和 Model 差不多，都是 AI。 | Agent 能串工具、反覆推理；Model 只負責單次輸入→輸出。選錯層會誤判技術需求。 |
| AutoML 可以自動化整個 ML 流程。 | AutoML 核心是「降低技術門檻」，讓非專家也能訓練圖片／文字／表格模型；全流程自動化是 MLOps 的範疇。 |
| 微調模型一定比現成 App 更好。 | 微調更精準但耗時耗資源；時間緊迫的草稿任務，Gemini for Workspace 才是最高效率。 |
| 邊緣 AI 只是「不用雲端」。 | 邊緣 AI 的核心價值是低延遲、隱私、離線——自動駕駛、手機端 Gemini Nano 才是典型場景。 |

### 第一部分：生成式 AI 五大層面 | The Five Layers

「生成式 AI 生態系統」由五個層面構成。**應用程式只是冰山一角**——使用者看得到的只有最上層，下面四層才是領導者要理解的「幕後技術與環境」。

由下而上（地基 → 使用者）：

```text
┌─────────────────────────────────────┐
│  ⑤ 應用程式層 Application          │  ← 使用者直接互動（Gemini App、Workspace、NotebookLM）
├─────────────────────────────────────┤
│  ④ 代理層 Agent                    │  ← 觀察、推理、採取行動（客服、程式碼、資料代理）
├─────────────────────────────────────┤
│  ③ 平台層 Platform                 │  ← 串聯各層、簡化 ML 工作流程（Vertex AI）
├─────────────────────────────────────┤
│  ② 模型層 Model                    │  ← AI 大腦（LLM、擴散模型等）
├─────────────────────────────────────┤
│  ① 基礎架構層 Infrastructure       │  ← GPU/TPU、儲存、網路
└─────────────────────────────────────┘
```

| 層面 | 英文 | 比喻 | 核心職責 | 代表產品 |
|------|------|------|----------|----------|
| **基礎架構** | Infrastructure | 地基與骨幹 | 運算（GPU/TPU）、儲存、網路；支撐訓練與部署 | GPU、TPU、儲存、全球光纖網路 |
| **模型** | Model | 大腦 | 從資料學習模式，生成內容、翻譯、回答問題 | Gemini、Claude、開源模型 |
| **平台** | Platform | 橋樑 | API、資料管理、模型部署；串聯模型與基礎架構 | Vertex AI |
| **代理** | Agent | 駕駛 | 自主設定目標、分析、決策；處理多步驟複雜工作 | 客服／程式碼／資料代理 |
| **應用程式** | Application | 前端車輛 | 使用者介面；直接與 AI 互動 | Gemini App、Workspace、NotebookLM |

**領導者金句**：不能只看到應用程式，必須理解五層如何連結，才能做出明智決策、促進創新並發揮 AI 最大潛力。

> 💡 **由上而下 vs 由下而上的理解順序**：使用者由上往下「用」AI（從 App 進入）；但工程師與決策者要由下往上「建」AI（先有算力地基，才有模型、平台、代理、應用）。理解這個方向差異，才不會把「換一個聊天機器人」誤當成「導入 AI」。

### 第二部分：代理（Agents）深度解析 | Agents Deep Dive

> **代理 = 一種特殊的應用程式**，能觀察情境、推論複雜概念、採取行動，並運用手邊的工具達成特定目標。

#### 1. 代理 vs 應用程式 | Agent vs Application

| 角色 | 職責 | 一句話 |
|------|------|--------|
| **代理 Agent** | 提供智慧與自動化；觀察情境、推理、採取行動、運用工具達成目標 | 提供**智慧** |
| **應用程式 Application** | 提供架構與外殼；定義整體目標、使用者介面、運作脈絡 | 提供**架構與目標** |

- 一個應用程式可包含**多個代理**（各自專長不同）→ **多代理系統（Multi-agent system）**
- 例：旅遊預訂 App（裡面有訂票、搜尋、推薦等多個代理）、客戶服務 App

#### 2. 代理兩大類型 | Two Types of Agents

| 類型 | 宗旨 | 流程 / 範例 |
|------|------|-------------|
| **對話式代理 Conversational** | 理解使用者意圖，以自然語言回應 | 接收輸入 → 理解意涵 → 呼叫工具（搜尋網路、存取資料庫） → 生成並提供回覆 |
| **工作流程代理 Workflow** | 自動化執行多步驟的複雜工作流程 | 電商訂單履行、新客入門流程、自動化研究（Gemini Deep Research）、資安紀錄剖析 |

#### 3. 代理 vs 獨立模型 | Agents vs Standalone Models

| 比較維度 | 獨立模型 (Models) | 生成式 AI 代理 (Agents) |
|----------|-------------------|-------------------------|
| 定位 | AI 大腦與基礎 | 問題解決者與任務執行者 |
| 組成 | 演算法 + 受訓資料 | **模型 + 推論迴圈 + 工具** |
| 運作 | 輸入 → 單次回應 | 觀察→解讀→規劃→行動（反覆循環） |
| 外部連結 | 無法直接操作外部系統 | 透過工具串接真實世界 |
| 適合任務 | 單次、線性內容生成 | 多步驟複雜工作流程 |
| 人為介入 | 需持續給指令 | 極少介入下自動完成 |

> **核心金句**：「AI 代理雖然使用模型，但能力遠遠超越模型。」獨立模型像具備知識的大腦；代理則替這個大腦裝上「思考迴圈（推論）」與「手腳（工具）」，使其能主動在真實世界完成複雜工作。

**推論迴圈（Reasoning Loop）**——代理的「思考過程」，是持續循環的反覆運算：

| 步驟 | 英文 | 做什麼 |
|------|------|--------|
| ① 觀察 | Observe | 收集環境與任務資訊 |
| ② 解讀 | Interpret | 處理資訊、評估當前情況 |
| ③ 規劃 | Plan | 擬定達成目標的行動方針 |
| ④ 採取行動 | Act | 執行規劃的動作 → 回到 ①，直到達成目標或停止點 |

> 此迴圈通常搭配 **CoT（Chain-of-Thought）**、**ReAct** 等提示架構引導推理。

**工具（Tools）**——代理與環境互動的必備功能，讓它連結真實世界的資訊與服務：

| 工具類型 | 範例 |
|----------|------|
| 資訊存取 | 搜尋網路、查詢資料庫、讀取公司庫存／日曆 |
| 系統操作 | 呼叫 API、寫入資料、觸發其他軟體 |
| 實體互動 | 操作實體機器人、控制裝置 |

#### 4. 常見代理使用情境 | Common Agent Use Cases

- 客服代理（Customer Service）、員工效率代理（Employee Productivity）、創意代理（Creative）
- 程式碼代理（Code）、資料代理（Data）、資安防護代理（Security）

### 第三部分：平台層 — Vertex AI | Platform Layer

**定位**：建構及擴充 AI 計畫的基礎與橋樑；介於模型、代理與基礎架構之間，整合基礎架構、工具、預訓練模型，簡化整個機器學習工作流程，協助開發人員與資料科學家加速推動 AI 計畫。

| 特點 | 說明 |
|------|------|
| 開放性與靈活性 | 支援多種框架與整合方式 |
| 強大基礎架構 | 可擴充、可靠的 Google Cloud 資源 |
| 預先訓練模型 | 現成模型資源 |
| 全方位工具 | 開發到部署完整工具鏈 |
| 自訂能力 | 依需求調整模型 |
| 輕鬆整合 | 透過 API 整合至應用程式 |

#### MLOps（Machine Learning Operations）

| 項目 | 內容 |
|------|------|
| **本質** | 工程與管理流程；類似 DevOps，但針對 ML |
| **核心功能** | **自動化、標準化、管理**機器學習專案**生命週期** |
| **涵蓋範圍** | 端對端工作流程、實驗、模型管理與調整、中繼資料追蹤、品質監控與評估 |
| **Feature Store** | 共用、提供、重複使用 ML 特徵，確保資料一致性 |

> ⚠️ MLOps ≠ 代理的推論迴圈。MLOps 管「運維」；推論迴圈管「思考與行動」。

### 第四部分：模型層 — 在 Vertex AI 選擇與建立模型 | Model Layer

**定位**：AI 系統的核心「大腦」。並非一般演算法，而是透過大量資料訓練而成的**複雜數學結構**，能學習資料中的模式與關係，進而生成內容、分析資料與分類資訊。

#### 兩大路徑 | Two Paths

| 路徑 | 適合對象 | 說明 |
|------|----------|------|
| **Model Garden** | 不想從頭開發 | 探索、自訂、部署 **160+ 現成模型**（Google 第一方、第三方如 Claude、開源模型）；可微調後部署 |
| **Model Builder — 自訂模型** | 資深 AI 研究人員 | PyTorch、TensorFlow、scikit-learn、XGBoost 等任何 ML 架構；大規模建立並**完全掌控**架構與訓練 |
| **Model Builder — AutoML** | ML 專業知識有限的團隊 | **僅需最基本技術知識**，以最少程式碼建構與訓練模型（低程式碼） |

#### Model Garden 的三類模型 | What's in the Garden

> 既然是「花園」，裡面種的就是一朵朵現成的「模型」。核心價值：不必從頭開發，直接進去**探索 → 自訂 → 部署**。

| 類別 | 內容 |
|------|------|
| **第一方基礎模型與預訓練 API** | Google 自家模型，以及整合最先進技術的 **Speech-to-Text（語音轉文字）**、**Natural Language Processing（自然語言處理）**、**Translation（翻譯）**、**Vision（電腦視覺）** API |
| **第三方模型** | 例如 Anthropic 的 **Claude** 模型系列 |
| **開放式模型** | 開放原始碼（Open-source）模型 |

- **靈活運用**：部分預訓練模型可直接使用；機構也能用自有資料**微調**後，部署至自訂應用程式。

#### AutoML 支援的資料類型

| 資料類型 | 任務 |
|----------|------|
| 圖片 | 分類、物件偵測 |
| 影片 | 動作辨識、分類、物件追蹤 |
| 文字 | 分類、實體擷取、情緒分析 |
| 表格 | 分類／迴歸、預測 |

#### 案例：Cymbal Aerospace（製造業）

- **痛點**：從龐大 3D CAD 模型擷取資訊耗時
- **做法 1**：AutoML 訓練專用模型，自動擷取材料、尺寸、公差等中繼資料
- **做法 2**：Vertex AI Pipelines + Gemini，將擷取資料轉為自然語言摘要
- **成效**：產品開發週期縮短 15%、設計審查時間減少 20%

> ⚠️ Model Garden = 探索／客製化／部署**現有模型**的服務，不是管基礎架構維運的平台。

### 第五部分：基礎架構層 | Infrastructure Layer

**定位**：建構任何 AI 系統的實體與軟體地基；結合硬體與軟體，提供訓練、部署、擴充 AI 模型（尤其是大型語言模型）所需的運算、儲存、網路。

| 三大支柱 | 說明 |
|----------|------|
| **高效能運算** | GPU、TPU；多節點超級電腦架構，大幅縮短訓練時間 |
| **高效能儲存** | 處理 PB 級資料集；容量與讀取速度缺一不可 |
| **網路** | 高頻寬、低延遲；Google 全球光纖網路確保叢集內資料流暢 |

**領導者金句**：基礎架構決定 AI 計畫的擴充性與成本效益。

**辨識口訣**：

- 運算力、伺服器、硬體、儲存空間 → **基礎架構層**
- 定義 AI 行為、多步驟自動化任務 → **代理層**

### 第六部分：邊緣 AI（Edge AI）| Edge AI

**定義**：在裝置本機或靠近資料來源的伺服器執行 AI，而非全部傳回遠端雲端。最大價值在於**即時回應、縮短延遲**。

| 優勢 | 說明 | 典型場景 |
|------|------|----------|
| **速度（低延遲）** | 消除雲端往返時間 | 自動駕駛避障、無人機 |
| **隱私權** | 資料在本機處理 | 醫療即時分析 |
| **離線存取** | 減少網路依賴 | 無網路環境仍可運作 |

| Google 工具 | 用途 |
|-------------|------|
| **LiteRT (Lite Runtime)** | 邊緣裝置／手機高效能 AI 執行階段 |
| **Gemini Nano** | 專為手機與嵌入式系統設計的精巧基礎模型（如 Pixel 通話筆記） |
| **AI Edge SDK** | Android 應用程式調用裝置端 AI |
| **Vertex AI** | 雲端端負責訓練、微調、MLOps；再轉換為 LiteRT 格式部署至邊緣 |

| 部署選擇 | 適合情境 | 典型案例 |
|----------|----------|----------|
| **邊緣 Edge** | 需瞬間即時反應、無法承受延遲、或需高度隱私 | 醫療手術即時分析裝置、自動駕駛汽車避障 |
| **雲端 Cloud** | 處理海量數據、高度集中化、需龐大可擴充算力 | 大型電商客服聊天機器人、智慧城市交通模式分析 |

> 🔑 **辨識口訣**：需要「即時、離線、隱私」→ 邊緣；需要「海量、集中、可擴充算力」→ 雲端。**Gemini Nano** 正是為資源受限的邊緣裝置與智慧型手機最佳化的精巧基礎模型（如 Pixel 通話筆記）。

### 第七部分：專案資源與可行性評估 | Resources & Feasibility

#### 1. 三大資源維度 | People · Cost · Time

**人力（角色）**

| 角色 | 職責 |
|------|------|
| 企業領袖／業務使用者 | 運用 AI 提升日常效率與流程 |
| 開發人員 | 用工具與 API 建構自訂 AI 方案；**部署代理並整合至 App** |
| AI 從業人員 | 模型自訂、部署、最佳化；Vertex AI、BigQuery、偏誤偵測、負責任 AI |

**費用（成本結構）**

| 活動 | 計費要點 |
|------|----------|
| 訓練模型 | 運算時間 + 訓練資料儲存 |
| 部署模型 | 端點費用 + 輸出儲存 |
| 使用模型 | 依用量計費（Tokens／字元）；模型越大通常越貴 |

**時間（自訂程度決定時程）**

| 方案類型 | 預估時間 |
|----------|----------|
| 預先建構 AI 應用程式 | 幾秒鐘即可開始 |
| 自訂 AI 代理 | 比從頭建模型快很多 |
| 專屬自訂模型（從頭） | 可能數個月 |

> **核心原則**：自訂程度越高 → 人力、成本、時間越多。

#### 2. 可行性評估六大維度 | Feasibility Dimensions

> 對應官方複習摘要「開始生成式 AI 專案前，請先思考」的需求清單：規模、自訂程度、使用者互動、隱私、延遲、連線能力。

| 維度 | 英文 | 評估要點 |
|------|------|----------|
| **規模** | Scale | 個人／小團隊用現成工具即可；百萬用戶需可擴充且安全的方案，並考量基礎架構成本、儲存與延遲 |
| **自訂程度** | Customization | 強基礎模型下未必需自訓；從現有模型著手，評估資料具體性與任務複雜度（文字摘要 vs 程式碼／創意寫作） |
| **使用者互動** | User Interaction | UI 整合方式（網站聊天小工具、嵌入現有 App）、UX（對話／資訊／任務導向）、所需指引與回饋程度 |
| **隱私權** | Privacy | 資料敏感程度、加密、存取控管、GDPR／HIPAA 等合規要求 |
| **延遲** | Latency | 回應速度要求；瞬時反應的情境傾向邊緣部署 |
| **連線能力** | Connectivity | 是否需離線運作；連線品質影響雲端／邊緣選擇 |

> 其他延伸考量：準確度與可解釋性（醫療／金融尤其重要）。

#### 3. 持續維護（旅程而非終點）

- 模型監控與重新訓練
- 資料更新
- 軟體更新與錯誤修正
- 硬體與基礎架構
- 安全性與法規遵循

| 行銷活動需「快速」生成產品說明 | 微調 Model Garden 模型 | **Gemini for Workspace 等現成應用程式** | 緊急 → Application |

| Gemini Nano 設計用途 | （其他） | **手機與邊緣裝置的精巧高效模型** | Nano = 裝置端 |
