# 生成式 AI 代理：實現組織轉型

> 生成式 AI 領袖學習路徑 · 第五堂課（最後一門）學習筆記

## Insight

生成式 AI 代理不是「更聰明的聊天機器人」，而是**模型 + 工具 + 推論迴圈**三者協同的系統。模型是大腦、工具是雙手與感官、推論迴圈（ReAct／CoT）是決策流程。**工具分類的關鍵不在名稱，而在「讀還是寫、靜態還是動態」**——資料儲存庫是圖書館（唯讀查閱），外掛程式是連到特定 App（日曆、Slack），擴充功能／函式才是動態運算與 API 呼叫。**對抗幻覺靠 Grounding，不靠多模態**——前者把答案錨定在可驗證來源，後者只是擴展輸入／輸出媒介。

> **一句話記法**：大腦（模型）→ 思考迴圈（ReAct／CoT）→ 手腳（工具四類）→ 知識庫（RAG／Vertex AI Search）→ 落地（多代理／Enterprise）→ 策略（願景＋指標＋變革）。

## Translation

| 原文 / 課程說法 | 我的理解 |
|----------------|----------|
| 確定性代理（傳統） | 照預先定義路徑走，不懂真實意圖——像自動販賣機選項。 |
| 生成式 AI 代理 | 理解意圖、存取外部資訊、自然互動——像有判斷力的助理。 |
| 基礎模型 (Foundation Model) | 代理的大腦；選模型要看訓練資料是否貼近用途。 |
| 工具 (Tools) | 代理與外部世界的橋樑——API、函式、資料庫、外掛。 |
| 推論迴圈 (Reasoning Loop) | 反覆「想 → 做 → 觀察 → 調整」的決策流程。 |
| ReAct (Reason and Act) | 思考 + 行動交替；適合需要查資料、呼叫 API 的任務。 |
| CoT (Chain of Thought) | 逐步推理、大聲說出思考過程；適合數學／邏輯難題。 |
| 元提示 (Meta-prompting) | 用 AI 幫你寫提示詞——先建「提示詞專家」，再生成系統指令。 |
| 擴充功能 (Extensions/APIs) | 標準化連外部服務；動態呼叫、運算。 |
| 函式 (Functions) | 封裝特定動作或邏輯（如 calculate_price）。 |
| 資料儲存庫 (Data Stores) | 靜態／唯讀知識庫——產品手冊、歷史紀錄；**只能查，不能寫**。 |
| 外掛程式 (Plugins) | 連到特定應用（日曆、Gmail、Slack）讀寫互動。 |
| RAG 三階段 | 檢索 → 增強提示詞 → 生成；可加疊代修正查詢。 |
| Grounding（建立基準） | 強迫 LLM 依第一方資料回答——對抗幻覺的「錨」。 |
| 混合式代理 (Hybrid Agents) | 確定性 AI 管業務邏輯 + 生成式 AI 管對話彈性。 |
| Gemini Enterprise | 企業內部 AI 中央平台——跨系統搜尋、管理代理、自動化工作。 |
| Agentspace | 整合內部資料的自訂搜尋與對話式代理（課程摘要用語；對應 Gemini Enterprise 脈絡）。 |

## Why

| 以前覺得 | 現在覺得 |
|----------|----------|
| 安排會議要查日曆 → 選資料儲存庫。 | 日曆互動是**外掛程式**；建議時段需要**擴充功能／函式**做動態運算；資料儲存庫只管靜態知識查閱。 |
| 減輕幻覺 → 多模態搜尋更強所以更準。 | 多模態只擴展媒介（文字／圖／影音），**不保證不瞎編**；抗幻覺靠 **Grounding** 綁定可驗證來源。 |
| RAG 讓模型永久學會新知識。 | RAG 是**當次查詢**先檢索再生成，不是改寫模型權重；好處是即時、可引用，不是永久記憶。 |
| Google AI Studio 和 Vertex AI Studio 差不多。 | AI Studio = 原型／學習（免費額度、RPM 限制）；Vertex AI Studio = 生產環境（企業安全、彈性配額、計費）。 |
| CoT 和 ReAct 只能二選一。 | CoT 偏重**內部推理**；ReAct 偏重**外部互動**；實務常**結合使用**。 |
| 一個代理搞定所有事。 | 複雜商用情境用**多代理系統**分工；代理本身也可當另一代理的「工具」。 |

### 第一部分：代理簡史與三大核心元件

#### 代理演進

| 階段 | 特徵 | 限制 |
|------|------|------|
| 確定性代理（早期） | 預先定義固定路徑 | 無法理解真實意圖 |
| 無 RAG 的生成式代理 | 自然語言對話 | 知識受限訓練資料、易幻覺 |
| 搭配 RAG 的生成式代理 | 理解意圖 + 外部知識 | 現代企業級代理基礎 |

> 測驗排序（由舊到新）：確定性代理 → 沒有 RAG 的生成式代理 → 搭配 RAG 的生成式代理

#### 三大核心元件

```
生成式 AI 代理
├── 基礎模型 (Foundation Model)  … 大腦（Gemini 等 LLM）
├── 工具 (Tools)                 … 手腳（API、函式、資料庫、外掛）
└── 推論迴圈 (Reasoning Loop)    … 決策流程（ReAct、CoT）
```

### 第二部分：使用模型

#### 取樣參數

| 參數 | 調低 | 調高 |
|------|------|------|
| **Temperature** | 較確定、標準（問答、摘要） | 較創意、不可預測（創意寫作） |
| **Top-K** | 只從機率最高的 K 個字詞選 | — |
| **Top-P** | 從累積機率達 P 的最小字詞子集取樣 | — |
| **其他** | 符記數量（長度上限）、安全性設定 | — |

#### Google AI Studio vs Vertex AI Studio

| 屬性 | Google AI Studio | Vertex AI Studio |
|------|------------------|------------------|
| 定位 | 快速探索、測試參數 | 建構、訓練、部署（企業級） |
| 目標用戶 | 初學者、原型設計 | 專業人士、雲端開發人員 |
| 登入 | 標準 Google 帳戶 | Google Cloud 控制台 |
| 限制 | RPM/TPM 用量限制，非大規模生產 | 依用量計費，配額彈性，企業安全與法遵 |

### 第三部分：提示工程與推論迴圈

#### 代理層與元提示

在終端應用前加一層「代理層」，嵌入角色提示、明確指示、少量樣本。**元提示**：先讓 AI 當「提示詞撰寫專家」，再為特定企業生成系統指令。

#### ReAct vs CoT

| 框架 | 核心 | 迴圈 | 適合 |
|------|------|------|------|
| **ReAct** | 思考 + 行動 | Think → Act → Observe → Adapt | 需查資料、呼叫工具、動態解題 |
| **CoT** | 逐步推理 | 展示中間推理步驟 | 數學、邏輯、結構化問題 |

**CoT 進階元件**：自我一致性（多解法取最一致）、主動提示（模型反問釐清）、多模態思維鏈。

> 實務可結合：CoT 做深度推理 + ReAct 做外部互動。

### 第四部分：工具與應用程式建構

#### 四類工具（必考）

| 類型 | 本質 | 典型情境 | 關鍵判斷 |
|------|------|----------|----------|
| **擴充功能 (Extensions/APIs)** | 標準化連外部服務 | 訂機票 API、動態比對時段 | 需要**呼叫 API、動態運算** |
| **函式 (Functions)** | 封裝特定動作／邏輯 | calculate_price、時間差計算 | 需要**程式邏輯、精確計算** |
| **資料儲存庫 (Data Stores)** | 靜態／唯讀知識 | 產品手冊、公司政策、RAG 知識庫 | 只需**查閱**文件，不寫入、不執行 |
| **外掛程式 (Plugins)** | 連特定應用／平台 | Google Calendar、Slack、日曆讀寫 | 要與**特定軟體**互動（讀＋寫） |

#### 工具選擇四問法（情境題速查）

```
① 只是查公司靜態文件？        → 資料儲存庫
② 要連日曆／Gmail／Slack？    → 外掛程式
③ 要算邏輯、比對、呼叫 API？  → 擴充功能 或 函式
④ 不確定時：有「寫入／發送／計算」？ → 不是資料儲存庫
```

#### 工具組合範例：會議地點規劃工具

單一代理可**同時呼叫多個工具**完成一項任務。例如「自動推薦最方便的會議地點」：

| 工具 | 角色 |
|------|------|
| **Document AI API**（擴充功能） | 處理文件、辨識與會者地址 |
| **Google Maps API**（擴充功能） | 分析各地址位置，計算最便利的集合點 |

> 重點：複雜任務常是**多個擴充功能／API 串接**，由推論迴圈決定呼叫順序——這正是 ReAct「推論 → 行動 → 觀察 → 疊代」的實際展現。
>
> 補充：外掛程式可連到的 Google 服務範圍也很廣，如 **Google 地圖、Google Workspace、YouTube、Google 相簿**等，皆透過 API 存取。

#### 經典考題：安排團隊會議

| 需求 | 正確工具 | 原因 |
|------|----------|------|
| 存取成員日曆、發送邀請 | **外掛程式** | 連接日曆 App，可讀寫 |
| 建議最合適時段 | **擴充功能**（或函式） | 需動態比對空檔、運算，非靜態查閱 |
| ❌ 資料儲存庫 | 不適用 | 只能查靜態知識，無法發邀請或算時段 |

#### ReAct 搭配工具的四步迴圈

1. **推論（工具選擇）** — 分析任務，選最合適工具
2. **執行動作** — 呼叫工具並傳入輸入
3. **觀察結果** — 接收輸出作為新觀察
4. **疊代** — 依觀察重新推論，直到完成

#### 從代理到應用程式

| 整合方式 | 說明 |
|----------|------|
| Gemini API | 任何能發 HTTP 的環境（網頁、行動、嵌入式） |
| Google AI Studio | 開發初期產 API 金鑰測試 |
| Vertex AI Studio | 生產環境、企業驗證授權 |
| Cloud Run / Cloud Run functions | 雲端部署 |
| Apps Script / AppSheet | 免／低程式碼整合 |

#### 多代理系統 (Multi-Agent Systems)

- 複雜情境拆分專業代理（航班、飯店、景點各自負責）
- **代理可作為工具**：客服代理呼叫「情緒分析代理」評估滿意度
- 核心概念：多個專業 AI 代理**協同合作**達成複雜目標

### 第五部分：RAG、Vertex AI Search 與企業方案

#### RAG 運作

| 階段 | 說明 |
|------|------|
| 1. 檢索 (Retrieve) | LLM 用工具從外部來源取相關資訊 |
| 2. 增強 (Augment) | 將檢索結果併入提示詞 |
| 3. 生成 (Generate) | 產出有引述、較準確的回覆 |
| 4. 疊代（選用） | 修正查詢詞、換工具、向使用者釐清 |

#### RAG 四大資訊來源

| 來源 | 用途 |
|------|------|
| 資料儲存庫 | 企業內部結構化／非結構化資料 |
| 向量資料庫 | 語意相似度搜尋（embedding） |
| 搜尋引擎 | 即時網頁、新聞（如 Google Search API） |
| 知識圖譜 | 實體與關係的結構化事實 |

#### 支援的資料儲存庫類型

- **網站**：公開網頁最新動態
- **結構化資料**：表格、JSON、產品目錄（Vertex AI 可理解結構）
- **非結構化資料**：PDF、DOCX、HTML 等

#### Vertex AI Search

| 能力 | 說明 |
|------|------|
| 搜尋解決方案 | 跨 BigQuery、Cloud Storage 等索引與搜尋 |
| 推薦解決方案 | 依使用者行為與內容屬性個人化推薦 |
| 搜尋摘要 | 自動生成結果摘要、產品比較 |
| 問答與後續問題 | 自然語言問答 + 對話式追問 |
| **Grounding（建立基準）** | 以第一方資料／知識圖譜為回覆基礎 → **減輕幻覺** |
| 企業級 | 精細存取控管、搜尋分析、API/SDK 整合 |

> **考點**：問「減輕幻覺」→ 選 **Grounding**，不是多模態搜尋。

#### Customer Engagement Suite（顧客參與度）

建構於 **CCaaS（Contact Center as a Service）** 之上：

| 產品 | 角色 |
|------|------|
| 對話式代理 (Conversational Agents) | 與顧客自然互動的聊天機器人 |
| Agent Assist | 即時支援真人客服（建議回覆、知識庫、轉錄／翻譯、摘要） |
| 對話式洞察 (Conversational Insights) | 分析情緒、主題、常見問題 |

**混合式代理**：確定性 AI（嚴格業務邏輯）+ 生成式 AI（對話彈性）。

**案例 Cymbal Taxes**：對話式代理 + 生成式劇本 + Vertex AI 資料儲存庫 → NPS +15、電話量 -20%、郵件 -30%。

#### 實作：Google AI Studio 打造 Cymbal Manufacturing 聊天機器人

1. 新對話，模型選 Gemini 2.5 Flash
2. 元提示：系統指令設為「虛擬代理程式助理建構專家」
3. 輸入企業背景與規範，讓 AI 生成優化系統提示詞
4. 新對話貼上提示詞，測試打招呼、產品顏色、無關問題（天氣）
5. 生產環境建議改用 **Conversational Agents**（企業級安全、內部資料庫）

#### Gemini Enterprise vs NotebookLM Enterprise

| 屬性 | NotebookLM Enterprise | Gemini Enterprise |
|------|----------------------|-------------------|
| 用途 | 特定文件深度分析 | 全方位企業 AI 助理與統一搜尋 |
| 資料範圍 | 使用者上傳的特定文件 | 所有已連結的業務系統 |
| 功能 | 回答、總結、針對該來源產內容 | 自動執行工作、跨系統找資訊、管理多代理 |
| 對象 | 內部知識工作者 | 中央發布與管理平台 |

**財富管理顧問範例流程**：Gemini Enterprise 檢索投資報告 → 結合 NotebookLM 客戶筆記 → 風險工具分析 → 自動撰寫客戶郵件草稿。

### 第六部分：策略——提前規劃

#### 導入生成式 AI 六大步驟

1. **確立明確願景** — AI 在組織的長期定位
2. **優先處理影響力高的用途** — 從最大效益／痛點切入
3. **投資能力與推動組織變革** — 協助員工適應新工作模式
4. **評估並展現價值** — ROI、收益、降本、效率、顧客體驗、安全
5. **提倡負責任的 AI** — 安全與倫理
6. **持續調整** — 定期檢討策略、與社群互動、培養創新文化

#### 專案成功路徑（測驗考點）

```
定義重要指標 → 收集及分析資料 → 反覆修正及改善
```

| 階段 | 分數 | 主要盲區 | 校正後 |
|------|------|----------|--------|
| 測驗 1 | **80%（通過）** | 工具分類：會議安排誤選「資料儲存庫」 | 釐清四類工具的讀／寫／算／連 App |
| 測驗 2 | **83%（通過）** | Vertex AI Search 抗幻覺誤選「多模態搜尋」 | 鎖死 Grounding = 綁定可驗證來源 |

> 兩次盲區都指向同一主題：**把「功能更強」誤當成「更可靠」**。資料儲存庫能查知識但不代表能執行動作；多模態能處理更多媒介但不代表不會瞎編。

#### 盲區 1：工具分類——安排團隊會議（測驗 1 第 3 題）

| 誤解 | 正確 |
|------|------|
| 存取日曆、建議時段、發邀請 → 外掛程式 + **資料儲存庫** | 外掛程式 + **擴充功能**（或函式） |
| 資料儲存庫能放日曆資料 | 資料儲存庫是**靜態唯讀知識**（手冊、政策），不能寫入邀請或動態算時段 |
| 建議時段 = 查資料 | 建議時段 = **動態比對空檔、運算** → 擴充功能／函式 |

#### 盲區 2：Grounding vs 多模態（測驗 2 第 6 題）

| 誤解 | 正確 |
|------|------|
| 減輕幻覺 → **多模態搜尋** | 減輕幻覺 → **建立基準 (Grounding)** |
| 多模態 = 更準確 | 多模態 = 能處理文字／圖片／影片／音訊，**不保證事實正確** |
| Grounding 是進階搜尋功能 | Grounding = 強迫 LLM **只依第一方／信任資料**回答，每句有「根據」 |

#### 測驗 1 答對重點

| 考點 | 正確觀念 |
|------|----------|
| Google AI Studio vs Vertex AI Studio | AI Studio 易用原型；Vertex AI Studio 企業級建構部署 |
| 多代理系統 | 多個專業代理協同達成複雜目標 |
| Temperature | 控制輸出隨機性／創意度 |
| 推論迴圈 | 觀察、內部推論、決策的疊代反覆過程 |

#### 測驗 2 答對重點

| 考點 | 正確觀念 |
|------|----------|
| Gemini Enterprise | 內部員工集中存取 AI 代理、安全運用公司資料 |
| 混合式代理 | 確定性管邏輯 + 生成式管對話彈性 |
| RAG 中的 LLM | 主動查詢資料儲存庫並擷取資訊豐富回覆 |
| AI 導入規劃 | 定義指標 → 收集分析 → 反覆改善 |
| 代理三大組成 | 基礎模型 + 工具 + 推論迴圈 |

#### 工具與可靠性速查表

| 題目關鍵字 | 聯想答案 |
|------------|----------|
| 查產品手冊、公司政策 | 資料儲存庫 |
| 連日曆、Slack、Gmail | 外掛程式 |
| 算價格、比對時段、呼叫 API | 擴充功能／函式 |
| 減輕幻覺、有憑有據、可靠資訊 | Grounding |
| 文字＋圖片＋影片 | 多模態 |
| 多個專家分工合作 | 多代理系統 |
| 原型測試、學習 | Google AI Studio |
| 生產部署、企業安全 | Vertex AI Studio |

### 第八部分：概念心智圖（一頁掌握全課）

```
生成式 AI 代理：實現組織轉型
│
├─ ① 代理架構
│    ├─ 基礎模型   — 大腦（Temperature、Top-P/K）
│    ├─ 工具       — 擴充功能／函式／資料儲存庫／外掛程式
│    └─ 推論迴圈   — ReAct（外部互動）+ CoT（內部推理）+ 元提示
│
├─ ② 建構與部署
│    ├─ Google AI Studio    — 原型、API 金鑰
│    ├─ Vertex AI Studio    — 生產、企業級
│    ├─ 單代理 → 多代理系統 — 分工、代理當工具
│    └─ Conversational Agents / Cloud Run / AppSheet
│
├─ ③ 知識與可靠性
│    ├─ RAG       — 檢索 → 增強 → 生成（可疊代）
│    ├─ Vertex AI Search — 企業搜尋 + 推薦 + 問答
│    └─ Grounding — 綁定第一方資料（抗幻覺錨點）
│
├─ ④ 企業應用
│    ├─ Customer Engagement Suite — 對話式代理、Agent Assist、洞察
│    ├─ 混合式代理 — 確定性 + 生成式
│    └─ Gemini Enterprise — 內部跨系統 AI 中央平台
│
└─ ⑤ 組織轉型
     ├─ 願景 → 優先用途 → 能力投資 → 變革管理
     ├─ 指標 → 資料 → 反覆改善
     └─ 認證測驗（Generative AI Leader）
```

### 第九部分：詞彙表（Glossary）

| 詞彙 | 一句話定義 |
|------|------------|
| **Agent（代理）** | 模型 + 工具 + 推論迴圈組成的自主決策系統。 |
| **Foundation Model** | 代理的大腦，如 Gemini。 |
| **ReAct** | 推理與行動交替的推論框架。 |
| **CoT（思維鏈）** | 逐步展示推理過程的提示技巧。 |
| **Meta-prompting（元提示）** | 用 AI 生成或優化提示詞。 |
| **Extensions（擴充功能）** | 透過 API 連外部服務。 |
| **Functions（函式）** | 封裝可執行的特定動作或邏輯。 |
| **Data Stores（資料儲存庫）** | 靜態唯讀知識來源，供查閱與 RAG。 |
| **Plugins（外掛程式）** | 連接特定應用（日曆、通訊工具）。 |
| **RAG** | 檢索增強生成——先查再答。 |
| **Grounding（建立基準）** | 把回覆錨定在可驗證來源，降低幻覺。 |
| **Multimodal（多模態）** | 同時處理文字、圖片、影音等媒介。 |
| **Multi-Agent System** | 多個專業代理協同完成複雜任務。 |
| **Hybrid Agent（混合式代理）** | 確定性邏輯 + 生成式對話。 |
| **Vertex AI Search** | 企業級搜尋、推薦與 RAG 方案。 |
| **Gemini Enterprise** | 企業內部 AI 代理中央管理與跨系統搜尋平台。 |
| **CCaaS** | Contact Center as a Service，雲端原生客服中心。 |
| **Temperature** | 控制模型輸出隨機性／創意度的參數。 |
