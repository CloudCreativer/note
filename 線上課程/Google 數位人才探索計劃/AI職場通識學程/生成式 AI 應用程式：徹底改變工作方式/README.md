# 生成式 AI 應用程式：徹底改變工作方式

> 生成式 AI 領袖學習路徑 · 第四堂課學習筆記

## Insight

職場導入生成式 AI 的關鍵不是學程式，而是**選對工具、寫對提示詞、綁對資料來源**。**Gemini 生態系各有分工**：Workspace 管日常辦公、Gem / Saved Info / 提示詞鏈結管個人化工作流、NotebookLM 管「深度讀特定檔案」、Cloud 管開發與企業部署。**Grounding 與 RAG 的本質是限縮與查證**——不是讓 AI 記更多或上網亂抓，而是把答案綁在可驗證的特定來源上，降低幻覺。

> **一句話記法**：怎麼問（提示詞技巧）→ 重複用（鏈結／Saved Info／Gem）→ 綁來源（Grounding／RAG／NotebookLM）→ 進企業（Gemini for Cloud）。

## Translation

| 原文 / 課程說法 | 我的理解 |
|----------------|----------|
| 零樣本提示：沒有先例就完成工作。 | 直接下指令，全靠模型內建知識——適合簡單、常見任務。 |
| 單樣本提示：提供一個範例給模型學習。 | 給「一個」示範，模型照樣造句；任務不難但要固定格式時用。 |
| 少量樣本提示：提供數個範例給模型學習。 | 給「幾個」示範讓模型抓到規律；任務較複雜、要精準模式時用。 |
| 角色提示：指派角色，調整風格、語氣和重點。 | 不是改模型，是改「誰在回答」——家教、分析師、客服語氣完全不同。 |
| 提示詞鏈結：在同一聊天中持續對話以維持脈絡。 | 像同一場會議延續討論；Gemini 記得前文，不必每次重貼背景。 |
| Saved Info：儲存特定資訊供模型持續使用。 | 全域設定——職稱、公司基本資料；**所有對話**都套用，不適合頻繁切換文風。 |
| Gem：個人化 AI 助理，依特定指示提供回覆。 | 專屬分身——「社群小編 Gem」「官方文案 Gem」各自獨立，切換文風的最優解。 |
| 建立基準：將輸出連至可驗證的資訊來源。 | 打官司要看證據——答案必須跟「你提供的特定文件」綁在一起，不是讓 AI 自由上網。 |
| RAG：先檢索外部知識，再生成回答。 | 外掛搜尋引擎——回答前先翻指定知識庫，讓輸出有憑有據、較不易過時或瞎編。 |
| NotebookLM：專屬研究助理，只根據上傳來源回答。 | 虛擬大腦擴充——讀報告、找矛盾、出大綱；**不做**簡報投影片、**不做**日曆排程。 |
| Gemini Code Assist：AI 程式設計夥伴。 | 結對編程——建議／生成程式碼、解釋邏輯，提升團隊編碼效率。 |

## Why

| 以前覺得 | 現在覺得 |
|----------|----------|
| NotebookLM 可以幫我做簡報、發行事曆。 | NotebookLM 核心是閱讀理解與研究——總結、提問、找關聯；做 Slides 用 Workspace，排程用 Calendar。 |
| RAG 是讓 AI 記住更多東西。 | RAG 是「回答前先查外部資料庫」——重點是準確與可驗證，不是擴充長期記憶。 |
| 切換寫作風格用 Saved Info 最方便。 | Saved Info 是全域單一背景；要部落格／社群／官網等多文風，應建**多個 Gem**。 |
| Grounding = 讓 AI 能上網查任何資料。 | Grounding 恰恰相反——**限縮**在特定、可驗證的來源（如公司 PDF），提高可靠性而非擴大搜尋範圍。 |
| Grounding 能讓 AI 從互動中學習並隨時間改善。 | Grounding 是靜態綁定來源，不是線上學習；它讓答案有跡可循，無法 100% 消除不準確。 |
| 零樣本／單樣本／少量樣本差在「樣本內容」。 | 差在**範例數量**：0 個、1 個、多個；數量越多越能鎖定你要的格式與模式。 |
| 用好 AI 一定要會寫程式。 | 本課程主打 No-code：Workspace、Gemini、Gem、NotebookLM、AppSheet 都能無程式碼落地。 |

## Action

- [ ] 為常用工作流建立 2～3 個 Gem（例如：社群文案、技術說明、會議摘要），各附角色提示與風格指南
- [ ] 用 NotebookLM 上傳一份專案文件集，試問「跨文件矛盾點」與「討論大綱」
- [ ] 在 Gemini 設定 Saved Info（職稱、產業），驗證全域背景 vs 獨立 Gem 的差異

### 第三部分：提示詞技巧（附實際範例）

> 提示詞技巧的差異，最容易混淆的是**樣本數量**。以下用「情緒分類」這一類任務，把零／單／少量樣本放在一起對比，差異一眼就懂。技巧之間可**自由組合**（例如：角色提示 + 少量樣本 + 鏈結）。

#### 基礎技巧對照

| 技巧 | 英文 | 核心 | 適用情境 |
|------|------|------|----------|
| 零樣本 | Zero-shot | 不提供範例，直接下指令 | 簡單、模型已有足夠知識的任務 |
| 單樣本 | One-shot | 提供**一個**範例讓模型學架構 | 需要特定格式但任務不複雜 |
| 少量樣本 | Few-shot | 提供**多個**範例引導 | 複雜任務、需精準理解模式 |
| 角色提示 | Role Prompting | 指派特定職務或身分 | 調整風格、語氣、重點 |
| 提示詞鏈結 | Prompt Chaining | 在同一對話中逐步深化 | 複雜工作，每步以上一步輸出為基礎 |

#### 範例 1｜零樣本（Zero-shot）— 直接問，靠模型內建知識

```
把以下這段會議記錄整理成 5 點重點摘要：
[貼上會議記錄]
```

> 沒有給任何範例，完全仰賴基礎模型既有的能力。

#### 範例 2｜單樣本（One-shot）— 給 1 個示範鎖定格式

```
請依照範例格式，把產品特色改寫成一句行銷標語。

範例：
特色：電池續航 20 小時
標語：一次充電，整天不斷電。

換你：
特色：防水深度 50 公尺
標語：
```

> 一個範例就足以讓模型抓到「短句、訴求情境、不條列規格」的風格。

#### 範例 3｜少量樣本（Few-shot）— 給多個示範讓模型抓規律

```
判斷以下客服訊息的情緒（正面／中立／負面）：

「東西很好用，謝謝！」          → 正面
「請問什麼時候到貨？」          → 中立
「等了兩週還沒收到，很失望。」  → 負面
「包裝有點壓到，但商品沒事。」  →
```

> 多個範例同時界定了**分類標準**與**輸出格式**，模型較不會自由發揮。任務越微妙、邊界越模糊，越需要 Few-shot。

#### 範例 4｜角色提示（Role）— 改變「誰在回答」

```
你是一位充滿教學熱忱的高中物理家教。
請用生活化的比喻，向完全沒有基礎的學生解釋「什麼是電壓」。
```

> 同一個問題，指派「物理家教」會用比喻＋鼓勵語氣；指派「論文審稿人」則會嚴謹、挑語病。**角色提示的主要優點＝調整回覆的風格、語氣與重點。**

#### 範例 5｜提示詞鏈結（Prompt Chaining）— 在同一對話逐步深化

```
① 幫我列出這篇文章的 5 個核心論點。
② 針對第 3 點，幫我找出可能的反方論述。
③ 把上面的正反論點整理成一段 200 字的平衡摘要。
```

> 每一步都以上一步的輸出為基礎；因為在同一個對話串裡，Gemini 記得前文，**不必每次重貼背景**。

#### 測驗重點（生成式 AI 提升工作效率）

| 考點 | 正確觀念 |
|------|----------|
| 零樣本 | 完全仰賴基礎模型內建知識，不提供任何範例 |
| 角色提示 | 主要優點是調整回覆的風格、語氣和重點 |
| Google Meet | 即時翻譯字幕並標示說話者，打破跨語言交流隔閡 |
| Google 文件 | 改善寫作風格、文法、語氣，生成不同措辭以潤飾提案 |

### 第四部分：重複使用提示詞 — 三種管理策略

| 策略 | 定義 | 優點 | 限制／適用 |
|------|------|------|------------|
| **提示詞鏈結** | 同一聊天對話中持續互動 | Gemini 記住對話歷史，不需重複背景 | 可**命名、釘選**對話以便日後存取 |
| **已儲存的資訊 (Saved Info)** | 在 Gemini 設定中儲存個人／團隊背景 | 每次獨立對話仍保有固定身分（如職位） | **套用至所有對話**；不適合頻繁切換截然不同文風 |
| **Gem** | 依特定指示與資源打造的專屬 AI 代理人 | 預載文風、知識庫；範本與引導式互動；每次新對話獨立 | 切換「社群小編／官網文案／技術說明」→ 建多個 Gem |

#### 決策速查

```
想維持同一討論的脈絡       → 提示詞鏈結
想全域記住身分與基本背景   → Saved Info
想針對特定專業任務獨立運作 → Gem（可加角色提示 + 上傳文件作知識庫）
```

#### 建立一個自訂 Gem 的步驟（No-code）

1. **命名與定位**：例如「Cloudia 社群小編」。
2. **寫角色提示**：身分、語氣、目標受眾、禁忌（如：不用 emoji 開頭、不超過 3 句）。
3. **上傳知識庫**：品牌風格指南、過往貼文、產品說明 → 這就是 Gem 的 **grounding 來源**。
4. **給範本與範例**：示範一兩則理想貼文（等於內建 Few-shot）。
5. **測試與微調**：每次展開新對話內容獨立，不會與其他 Gem 互相外洩。

> **關鍵差異**：Saved Info 是「一個你」套用到所有對話；Gem 是「多個專家分身」各自獨立。多文風需求 → 一定是多個 Gem。

### 第五部分：建立基準（Grounding）與 RAG

#### 什麼是「建立基準 (Grounding)」？

| 項目 | 內容 |
|------|------|
| **定義** | 將 AI 輸出內容**連結至可驗證的特定資訊來源** |
| **No-code 做法** | 在 Gem 或 Gemini 提示詞中上傳檔案／提供資源（如公司風格指南） |
| **好處** | 更豐富情境 → 更準確、具洞察力、符合需求的結果；答案**有跡可循** |
| **不是** | ❌ 讓 AI 從互動中學習改善　❌ 100% 防止不準確　❌ 無限制上網搜尋 |

> **核心比喻**：打官司要看證據——沒有 grounding，AI 憑記憶與直覺回答（易幻覺）；有 grounding，答案必須跟「你提供的特定可驗證來源」綁在一起。

#### RAG：檢索增強生成 (Retrieval-Augmented Generation)

| 步驟 | 說明 |
|------|------|
| 1. 檢索 (Retrieval) | 用**語意搜尋／向量資料庫**，從外部知識庫擷取與查詢最相關的資訊 |
| 2. 生成 (Generation) | 模型結合擷取到的資訊，生成最終回答 |

**RAG 背後的運作（補充，幫助理解「語意搜尋」與「向量資料庫」）：**

```
文件 → 切成小段 (chunk) → 轉成向量 (embedding) → 存入向量資料庫
                                                        ↓
使用者提問 → 也轉成向量 → 在向量庫中找「語意最接近」的片段
                                                        ↓
            把找到的片段塞進提示詞 → 模型據此生成回答（附來源）
```

> 「語意搜尋」找的是**意思相近**而非字面相同，所以問「怎麼退貨」也能找到標題寫「退換貨政策」的段落。

#### 僅模型 vs 採用 RAG

| 對照 | 僅模型（不含 RAG） | 採用 RAG |
|------|-------------------|----------|
| 外部知識 | 無 | 有（文件、資料庫、網路等指定來源） |
| 時效性 | 可能過時 | 可依最新外部資料 |
| 幻覺風險 | 較高 | 較低（有來源可查） |
| 可解釋性 | 低 | 可顯示引用來源 |

#### RAG 核心優勢

1. **提升準確度與關聯性** — 直接運用相關資訊
2. **提升可解釋性與透明度** — 顯示特定來源，說法可經查證
3. **擴充 LLM 功能** — 限制模型只根據取得的情境生成回覆

> **免代碼 RAG 工具**：**NotebookLM**

#### 補充：Grounding vs RAG vs 微調 (Fine-tuning) — 別搞混

| 技術 | 做法 | 改變什麼 | 最適合 |
|------|------|----------|--------|
| **Grounding** | 把答案綁在你提供的來源 | 限縮「回答依據」 | 要求可驗證、降低幻覺 |
| **RAG** | 回答前先從外部知識庫檢索 | 動態補充**最新／專屬知識** | 知識量大、會更新 |
| **微調 (Fine-tuning)** | 用資料**再訓練模型權重** | 改變模型**本身行為／文風** | 固定任務、特定語氣，且需求穩定 |

> 直覺記法：Grounding／RAG 是「給模型看小抄」（不改模型）；微調是「重新教育模型」（改模型本身，成本高）。日常職場需求，先用前兩者。

### 第六部分：NotebookLM

#### 定位與版本

| 項目 | 內容 |
|------|------|
| **核心定義** | 以 AI 為主軸的虛擬筆記本與研究夥伴 |
| **Grounding 原則** | **只**根據使用者上傳的來源回答；超出範圍會誠實告知，**不憑空編造、不從廣泛網路擷取** |
| **NotebookLM Plus** | 更高容量與分析功能 |
| **NotebookLM Enterprise** | 企業隱私安全、**IAM 角色**控管筆記本存取 |

#### NotebookLM vs Gem vs Gemini

| 工具 | 擅長 | 不擅長 |
|------|------|--------|
| **Gem** | 一般性廣泛工作、日常工作流程、多文風切換 | 深度分析整包專屬文件 |
| **NotebookLM** | 深入瞭解、分析**特定專屬資訊**；跨來源找關聯與矛盾 | **不做簡報投影片、不做日曆排程、不從廣泛網路擷取** |
| **Gemini（一般）** | 通用對話、創意、規劃 | 無專案級文件 grounding 時易幻覺 |

#### 核心運作與用途

- **筆記本** = 專案容器；上傳 PDF、簡報、音訊、影片、網址等多元來源。
- **課程提到的能力**：自動總結重點、找多來源**關聯／矛盾**、生成大綱草稿、**製作測驗**驗收學習。
- **團隊協作**：共用筆記本，建立集體「大腦」，一起提問、掌握最新異動，培養知識分享文化。
- **適用情境**：研究新主題、製作訓練教材與說明文件、準備簡報大綱、整理會議重點、擬定專案提案。

> 📌 **補充（NotebookLM 實際還有的產出，課程未必逐一點名）**：語音摘要 Audio Overview（像 Podcast 對談）、學習指南、簡報文件 Briefing Doc、常見問答 FAQ、時間軸 Timeline、心智圖。核心定位仍是「讀懂你的來源」，而非取代 Slides／Calendar。

### 第七部分：Gemini for Google Cloud

#### 產品線總覽

| 產品 | 角色 | 核心能力 |
|------|------|----------|
| **Gemini Cloud Assist** | 雲端架構專家 | 設計／管理／調整應用；分析環境、指標、記錄檔；個人化優化建議 |
| **Gemini in BigQuery** | 資料分析大師 | 自然語言探索資料、編寫程式碼、自動生成洞察（含函式呼叫轉 SQL） |
| **Gemini Code Assist** | 結對編程夥伴 | 20+ 語言；建議／生成程式碼、直白解釋 |
| **Gemini in Colab Enterprise** | Python / ML 利器 | 互動式筆記本中自動建議與生成程式碼 |
| **Gemini in Databases** | DBA 助手 | 自然語言建構應用、單一介面管理資料庫機群 |
| **Gemini in Looker** | BI 特助 | 直覺理解資料、自動製圖、生成報表 |
| **Gemini in Security** | 資安防衛官 | 近乎即時分析攻擊路徑、安全發現、威脅情報彙整 |

#### 案例：Cymbal Retail

| 項目 | 內容 |
|------|------|
| **痛點** | 行銷團隊想分析 BigQuery 海量資料，極度依賴資料分析師 |
| **解法** | Gemini in BigQuery（含函式呼叫）— 自然語言提問 → 自動轉 SQL → 呈現結果 |
| **成效** | 行銷直接做資料導向決策；分析師專注高策略價值計畫 |

#### 企業級安全

| 要點 | 說明 |
|------|------|
| 資料隱私 | Google **不會**使用客戶提示詞或 Gemini 回覆來訓練模型 |
| 基本安全要素 | 客戶資料**加密** + **嚴格存取權控管** |
| 機構資料 | 不公開、不外洩給第三方開發人員 |

#### 測驗重點（Gemini for Google Cloud）— 100%

| 考點 | 正確觀念 |
|------|----------|
| Gemini Code Assist | 開發團隊的 AI 程式設計夥伴，協同編寫更優質程式碼，提升編碼效率與協作 |
| 企業安全模型 | 客戶資料皆經加密，並設有嚴格存取權控管 |

### 第八部分：測驗盲區複習（認知校正）

#### 我的測驗歷程（學習軌跡）

| 階段 | 分數 | 主要盲區 | 校正後 |
|------|------|----------|--------|
| 第一次（建立基準單元） | **33%** | 把 NotebookLM 當簡報／排程工具；誤解 Grounding 是「線上學習／防所有錯誤」 | 釐清 NotebookLM 邊界、Grounding 本質 |
| 第二次 | **83%（通過）** | 唯一扣分在「接地 Grounding」的核心定義 | 鎖死「綁定可驗證來源」這一句 |
| Gemini for Google Cloud | **100%** | — | Code Assist 定位、企業安全模型已掌握 |

> 進步軌跡 33% → 83% → 100%，盲區幾乎都集中在 **Grounding／RAG／NotebookLM 三者的邊界**。這三個是本課最容易混淆、也最常考的地方。

#### 盲區 1：NotebookLM 的邊界

| 誤解 | 正確 |
|------|------|
| 可自動生成含投影片的簡報、發送日曆邀請 | 核心是**閱讀理解與研究**——總結、找盲點、討論大綱；簡報用 Slides，排程用 Calendar |

#### 盲區 2：RAG 的目的

| 誤解 | 正確 |
|------|------|
| 提升 AI 記住大量資訊的能力 | **外掛搜尋**——回答前先查指定知識庫，讓回答有憑有據、符合現狀 |

#### 盲區 3：Gem vs Saved Info

| 誤解 | 正確 |
|------|------|
| 頻繁切換文風用 Saved Info | Saved Info = 全域單一背景；多文風 → **多個 Gem** |

#### 盲區 4：Grounding 的本質

| 誤解 | 正確 |
|------|------|
| 讓 AI 從互動中學習並隨時間改善 | 靜態綁定可驗證來源，**非線上學習** |
| 防止所有不準確回覆 | **降低**幻覺機率，無法 100% 保證 |
| 允許存取網際網路上任何網站 | **限縮並綁定**特定來源，不是無限上網 |

#### 工具選擇速查表

| 需求 | 推薦工具 |
|------|----------|
| 深入研究特定整包檔案 | NotebookLM |
| 回答依據最新外部資料 | RAG（或 NotebookLM） |
| 多種不同文風的專屬專家 | 多個 Gem |
| 全域固定身分背景 | Saved Info |
| 同一專案延續對話脈絡 | 提示詞鏈結（釘選對話） |
| 日常辦公（郵件、文件、會議） | Gemini for Workspace |
| 寫程式、查雲端、分析大數據 | Gemini for Google Cloud |

### 第九部分：概念心智圖（一頁掌握全課）

```
生成式 AI 應用程式：徹底改變工作方式
│
├─ ① 怎麼問（提示詞技巧）
│    ├─ 零樣本   — 0 範例，靠模型內建
│    ├─ 單樣本   — 1 範例，鎖格式
│    ├─ 少量樣本 — 多範例，抓規律
│    ├─ 角色提示 — 改風格／語氣／重點
│    └─ 提示詞鏈結 — 同對話逐步深化
│
├─ ② 重複用（個人化與工作流）
│    ├─ 提示詞鏈結 — 維持單一對話脈絡（可命名／釘選）
│    ├─ Saved Info — 全域固定身分背景
│    └─ Gem        — 多個專家分身（角色 + 知識庫）
│
├─ ③ 綁來源（可靠性）
│    ├─ Grounding — 把答案綁在可驗證來源（降幻覺）
│    ├─ RAG       — 回答前先檢索外部知識庫（語意搜尋 / 向量庫）
│    └─ NotebookLM — No-code RAG，只讀你上傳的來源
│
└─ ④ 進企業（Gemini for Google Cloud）
     ├─ Code Assist / BigQuery / Cloud Assist
     ├─ Colab / Databases / Looker / Security
     └─ 安全：資料加密 + 嚴格存取控管 + 不拿你的資料訓練
```

### 第十部分：詞彙表（Glossary）

| 詞彙 | 一句話定義 |
|------|------------|
| **Zero-shot（零樣本）** | 不給範例，直接下指令。 |
| **One-shot（單樣本）** | 給 1 個範例讓模型照樣造句。 |
| **Few-shot（少量樣本）** | 給多個範例讓模型抓規律。 |
| **Role Prompting（角色提示）** | 指派身分以調整風格、語氣、重點。 |
| **Prompt Chaining（提示詞鏈結）** | 同一對話中逐步深化，沿用前文脈絡。 |
| **Saved Info（已儲存資訊）** | 全域套用的個人／團隊背景設定。 |
| **Gem** | 依指示與知識庫打造的專屬 AI 分身。 |
| **Grounding（建立基準）** | 把輸出綁定到可驗證的特定來源。 |
| **RAG（檢索增強生成）** | 先檢索外部知識，再據此生成回答。 |
| **Embedding（向量嵌入）** | 把文字轉成數字向量，方便比對語意相近度。 |
| **Vector DB（向量資料庫）** | 存向量、做語意搜尋的資料庫。 |
| **語意搜尋** | 找「意思相近」而非字面相同的內容。 |
| **幻覺 (Hallucination)** | 模型自信地編造不正確資訊。 |
| **Fine-tuning（微調）** | 用資料再訓練模型權重以改變其行為。 |
| **NotebookLM** | 只根據上傳來源回答的 No-code 研究助理。 |
