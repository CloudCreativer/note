# 筆記整理規範

## 1. 目標

這份規範用來統一你的筆記資料夾命名、Markdown 內容結構與後續維護方式，讓資料更容易查找、閱讀與擴充。

## 2. 資料夾命名規則

格式：

```text
編號_公司名稱_關鍵字
```

範例：

```text
001_OpenAI_API-design
002_Apple_iOS-interview
003_Google_Product-strategy
```

命名原則：

- `編號` 固定使用三碼，從 `001` 開始。
- `公司名稱` 使用固定寫法，不要同一家公司混用不同名稱。
- `關鍵字` 用來描述這份筆記的主題，建議 1 到 3 個詞。
- 多個關鍵字請用 `-` 連接，不要用空白。
- 避免使用過長名稱，讓資料夾一眼就能辨識。

## 3. 建議資料夾結構

每一份主題筆記建議使用以下結構：

```text
001_公司名稱_關鍵字/
  README.md
  assets/
```

結構說明：

- `README.md`：該資料夾的主筆記，所有重點都整理在這裡。
- `assets/`：存放圖片、截圖、PDF 匯出圖或其他附件。

## 4. Markdown 撰寫規則

- 每個資料夾只維護一份主筆記：`README.md`。
- 標題層級固定使用 `#`、`##`、`###`，不要跳階。
- 先寫摘要，再寫細節，方便快速掃描。
- 待辦事項統一使用 Markdown checkbox。
- 相關連結集中在文末，避免內容四散。
- 原始資料與整理後結論分開，避免資訊混雜。

## 5. README.md 標準範本

以下內容可直接複製到每個資料夾內的 `README.md`：

```md
---
id: 001
company: 公司名稱
keyword: 關鍵字
status: inbox
created: 2026-05-27
updated: 2026-05-27
tags:
  - 公司名稱
  - 關鍵字
source:
  - 
---

# 001 公司名稱 關鍵字

## 一句話摘要

用 1 到 3 句話說明這份筆記的核心內容與結論。

## 背景

- 為什麼會有這份筆記
- 筆記的情境、來源或目的
- 這份資料預計拿來做什麼

## 重點整理

### 主題 1

- 重點 1
- 重點 2

### 主題 2

- 重點 1
- 重點 2

## 關鍵結論

- 結論 1
- 結論 2
- 結論 3

## 待確認問題

- [ ] 問題 1
- [ ] 問題 2

## 後續行動

- [ ] 行動 1
- [ ] 行動 2

## 相關連結

- [名稱](https://example.com)

## 原始資料

- 來源 1
- 來源 2

## 更新紀錄

- 2026-05-27：建立初版
```

## 6. 維護建議

- 新增筆記時，先建立資料夾，再複製 `README.md` 範本。
- 若同一家公司有多份主題，請保持公司名稱完全一致。
- 若主題延伸太大，新增新的 `編號_公司名稱_關鍵字` 資料夾，不要把多個主題塞進同一份筆記。
- 每次修改內容時，同步更新 `updated` 欄位與 `更新紀錄`。

## 7. 建議執行方式

之後可以依這個流程整理：

1. 建立資料夾：`編號_公司名稱_關鍵字`
2. 在資料夾內建立 `README.md`
3. 把附件放進 `assets/`
4. 先填「一句話摘要」與「關鍵結論」
5. 最後補齊背景、原始資料與後續行動

## 8. 活動現場筆記（簡報／逐字稿）

適用情境：創業競賽、Demo Day、投資簡報等，同時有簡報截圖與錄音逐字稿。

### 資料優先順序

1. **簡報截圖、現場手寫或圖片筆記**（數字、圖表、公司名以這裡為準）
2. **現場 Q&A 紀錄**（評審問題與團隊回答）
3. **自動逐字稿**（補充脈絡；專有名詞、人名、金額常有誤，需交叉比對）

### 公司 README 應寫什麼

- 只寫**讀得懂的結論**：產品在做什麼、解決誰的痛點、商業模式、關鍵數字、Q&A 重點
- Q&A 用評審原問 + 講者原答，格式如下（可略修逐字稿錯字，但不要改寫成摘要表）：

```md
## 評審提問與回答

### Q1. （評審問題原文或略修）

（團隊回答原文或略修）
```
- 不確定的數字或名稱放「待確認問題」，並標註來源（簡報／逐字稿／待查）

### 不要寫進公司 README 的內容

- 整理方法、撰寫規範、逐字稿誤差說明（寫在本檔 `NOTE_STANDARD.md`）
- 長篇「如何驗證」「建議追問清單」模板
- 與該公司無關的產業通論
